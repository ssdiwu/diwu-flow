## 不可协商规则

- 必须在检测到六类退化信号中任一信号时立即停止扩写，回到四行重写模板重新锚定
- 禁止在无运行态或输出层证据的情况下宣称「已完成」
- 禁止把副本/旧入口/旧文档当作当前主线继续推进
- 必须先排除六类泛化误判再下结论，不得跳过误判排查表直接归因
- 区分纠偏与 BLOCKED 时必须使用 DECISION TRACE 结构化输出

# dcorr

纠偏体系：偏航时的恢复协议。开工前的完整规范见 mindset（三唯一框架、不确定性门控）。

---

## 使用方式与触发条件

出现以下**任一退化信号**时，停止扩写，回到本规则。同时停掉对应的错误动作（右列）：

| # | 退化信号 | 含义 | 对应停止项 |
|---|---------|------|-----------|
| 1 | 同一问题被纠偏两次以上仍沿旧路径推进 | 判断循环 | 沿已被否定的路径继续扩写 |
| 2 | 同一前提被反复解释 | 上下文污染 | 把副本/旧入口/旧文档当当前主线 |
| 3 | 输出越来越像通用套路 | 目标漂移 | 把界面表象直接当主链根因 |
| 4 | 讨论越来越长，下一步动作越来越模糊 | 动作衰减 | 把测试红/提示成功直接当最终结论 |
| 5 | 已改动却拿不出运行态或输出层证据 | 证据缺失 | 无证据宣称"已完成" |
| 6 | 同时引用多个目录/入口/"真相源" | 主线分裂 | 把问题归因给模型/提示词/代码的单一层 |

## 四行重写模板

纠偏时先用四行重新锚定当前任务。**写不出来 = 还没回到主线**。

```text
当前目标：- 这次到底要得到什么结果
当前主线：- 唯一主线目录 / 唯一运行入口 / 唯一 canonical
当前现象：- 已观测到的事实（不是推测）
当前缺口：- 缺什么证据导致还不能推进
```

轻量任务（纯写作/低运行态）可用简化版：目标 + 主线 + 现象 + 下一步动作 + 验证方式。

## 误判排查表（六类泛化模式）

纠偏的第一步是排除误判。看到对应现象时，**先查左列再下结论**。

### 1. 环境漂移

| 项 | 内容 |
|----|------|
| **现象** | 今天突然不通、502/404/超时、同提交多机表现不一致 |
| **默认先查** | 运行进程、端口占用、代理配置、环境变量、依赖服务状态 |
| **暂不下结论** | "代码回归了"、"上次还好好的" |
| **何时下结论** | 排除所有环境因素后，同一代码在干净环境中仍复现 |

### 2. 数据缺口

| 项 | 内容 |
|----|------|
| **现象** | 查询没命中、结果为空、候选缺失、显示"证据不足" |
| **默认先查** | 测试样本量、账号权限、数据存量、索引状态 |
| **暂不下结论** | "实现失效了"、"算法有问题" |
| **何时下结论** | 数据充足且权限正确时仍复现 |

### 3. 读层现象

| 项 | 内容 |
|----|------|
| **现象** | 页面白屏、布局错位、打不开、时间离谱、幽灵数据 |
| **默认先查** | 前端读层逻辑、渲染管线、开发服务状态、CSS/JS 加载 |
| **暂不下结论** | "后端主链挂了"、"API 返回错了" |
| **何时下结论** | 前端独立验证通过（直接调 API 确认返回正确）后仍有问题 |

### 4. 路由/护栏/契约

| 项 | 内容 |
|----|------|
| **现象** | tool-loop-guard 触发、空查询、选错工具、XML/格式漂移 |
| **默认先查** | 查询参数构造、工具路由规则、接口格式约定、护栏触发条件 |
| **暂不下结论** | "模型能力不行"、"这个模型做不到" |
| **何时下结论** | 参数和路由完全正确时仍复现 |

### 5. 验证误读

| 项 | 内容 |
|----|------|
| **现象** | 测试失败/脚本失败/HTTP 200/成功提示/撤回成功文案 |
| **默认先查** | 测试口径和前提、脚本执行时机、表面提示含义、首轮输出完整性 |
| **暂不下结论** | "真实失败了" 或 "真实成功了" |
| **何时下结论** | 用 L1-L3 证据交叉验证后 |

### 6. 分层未拆清

| 项 | 内容 |
|----|------|
| **现象** | 结构化结果差、产出回退、改了没效果 |
| **默认先查** | 输入层 → 提示词层 → 后置判定层 → 回退层，逐层隔离 |
| **暂不下结论** | "提示词的问题" 或 "代码的问题"（单一归因） |
| **何时下结论** | 逐层排查后锁定单一层，且其他层独立验证正常 |

## 入口重判门控

纠偏后不要默认继续原路径。重新判断入口方式：

| 条件 | 路径 | 要求 |
|------|------|------|
| 改动小 + 结果可预期 + 差异一句话说清 | 直接继续 | 三项同时满足 |
| 任一不满足 + 需稳定输出/交接 | 先写最小规格 | 见 mindset 最小规格模板 |
| 外部依赖多 / 落点不清 / 回滚成本高 | 先探索验证 | 隔离验证后再决定 |

---

## 恢复后输出骨架

重新开始推进时，第一条输出**禁止宣称"已经搞定"**：

```
现象：- 我现在看到的事实
判断：- 基于事实我认为更像什么；已排除哪些误判
动作：- 下一步具体做什么；怎样验证
```

## 止损序列

尝试重写任务、确认主线、补事实后**仍回不到主线**时，按序止损。工具连续失败场景优先用 session.md 3-Strike 协议（1/3 诊断 → 2/3 换方法 → 3+ 升级用户）：

| 步骤 | 动作 | 目的 |
|------|------|------|
| 1 | 切掉已污染的历史上下文 | 清除判断噪声 |
| 2 | 把任务缩到最小可执行单元 | 降低复杂度 |
| 3 | 明确写出缺口（边界/证据/外部依赖） | 定位阻塞点 |
| 4 | 请求人工补充主线/验收边界/判别依据 | 外部介入 |

**最忌讳**："继续解释，希望解释本身把任务救回来"。

## 交接最小锚点

交接时不默认转发完整历史。只传五类信息：当前目标、关键边界、已验证结论（逐条 PASS/FAIL）、未解决问题、下一步动作。

---

## 无法验证与 BLOCKED 的边界

三者的区分判断：问"给我正确的判断就能继续吗？" → 能 = correction；缺环境/凭据 = BLOCKED；缺条件但未来可满足 = 无法验证。

| 维度 | correction（纠偏） | 无法验证 | BLOCKED（阻塞） |
|------|-------------------|---------|---------------|
| **根因** | AI 推理出错，内部判断失真 | 时间窗口/环境临时不可用 | 凭据缺失/服务停运/权限不足 |
| **解决方** | AI 按本协议恢复 | 条件具备后自行验证 | 必须等待人工提供新条件 |
| **任务状态** | 保持 InProgress | 保持 InProgress | 退回 InSpec |

### 无法验证三段式模板

```text
未验证项: [具体条目]
原因: [为什么现在无法验证]
后续验证方式: [具体步骤或条件，不是"以后再测"]
```

**禁止**：用"理论上应该没问题"替代验证 / 未验证项混入已验证项不加标注 / 因无法验证跳 acceptance 条目。
