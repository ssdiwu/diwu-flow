# didea

想法的挂载点、持久化容器、下游能力连接器。**不是诊断器，不是清单生成器，不是 PRD 工厂。**

---

## 定位：第 0 层入口容器层

```
灵感 / 对话片段
  ↓
didea（挂载到 .diwu/ideas/）
  ↓
后续可进入 dpth(判断) / dref(细化) / dprd(论证) / dtask(任务化)
```

didea 只负责一件事：**让想法不丢失，并为它找到下一步的去向。**

### 它不是什么

| 维度 | didea | 不是这个 |
|------|-------|---------|
| 判断 | 入口容器 | dpth（产品诊断） |
| 细化 | 挂住想法 | dref（可执行清单） |
| 论证 | 连接下游 | dprd（PRD 文档） |
| 执行 | 不走 drun | dtask + drun |
| 设计 | 容器层 | architect（技术设计） |

### 与兄弟 skill 的交互对比

| skill | 交互方式 | 产出 | 门槛 | 与 didea 关系 |
|-------|---------|------|------|--------------|
| **dref** | AI 主动提问（推式） | 对话中检查清单 | "想把这个想清楚" | 下游：didea → 可选 dref |
| **dtask** | 手动结构化填写 | dtask.json 条目 | "要开始做了" | 下游：didea refine → dtask |
| **dprd** | 正式文档撰写 | 完整 PRD 文档 | "值得正式设计" | 下游并行：大需求走 dprd |
| **didea** | 单次快速捕获 | `.md` 文件 + 可选 Issue | "先记下来" | **入口** |

## 双入口模型

### 软入口（SKILL.md 自然识别）

在对话中识别到「这是一个值得挂起来的 idea」时：

1. **主动提议**：「这听起来像一个值得记录的想法，要不要挂到 `.diwu/ideas/`？」
2. **解释去向**：挂入后可以接 `dpth` 做判断、`dref` 做细化、`dprd` 写 PRD、`dtask` 拆任务
3. **等待确认**——**不自动落盘**

#### 识别信号（≥4 种对话模式）

| 信号模式 | 典型话术 | 触发条件 |
|---------|---------|---------|
| 突然冒出新方向 | "对了，其实还可以做 XXX" | 对话中途插入与当前主题无关的新方向 |
| 明确要求暂存 | "这个先记下来"、"先挂着" | 用户显式表达暂存意图 |
| 未来式表达 | "以后可能要做"、"等有空了再考虑" | 时间上不确定但认为有价值 |
| 灵感爆发式 | "我有个想法！"、"突然想到" | 高能量但未收敛的灵感表达 |

#### 提议话术模板

```
[捕捉到信号] → 「这听起来像是一个值得记录的想法。
 要不要我先把它挂到 .diwu/ideas/？
 挂进去之后随时可以用 /didea list 查看，
 也可以后续推给 dpth 做方向判断或 dref 做细化。」
```

### 硬入口（commands/didea.md 显式操作）

用户明确调用 `/didea <动作>` 时的稳定操作入口。见下方「六个硬入口动作」章节。

## 动作门控：三条硬规则

> 这是 didea 的安全底线。违反任何一条都是越界。

### 规则 1：识别可以自然发生

软入口的识别是持续监听行为，不需要用户显式触发。

| 类型 | 场景 | 判定 |
|------|------|------|
| 正例 | 对话中用户说"对了还可以做个 XX"，AI 识别后提议挂入 | 合规，这是软入口的核心价值 |
| 反例 | 用户没表达任何想法，AI 主动说"你有没有什么新想法要记？" | 违规，不要制造需求 |
| 边界 | 用户在讨论已有任务时顺带提了一句"以后可以扩展"，温和提议即可 | 合规，低打扰提议 |

### 规则 2：本地写入必须用户确认

无论软入口还是硬入口，写入 `.diwu/ideas/` 前必须获得用户明确确认。

| 类型 | 场景 | 判定 |
|------|------|------|
| 正例 | AI 提议后用户回复"好，记下来"，然后才写文件 | 合规 |
| 反例 | AI 识别到想法后直接创建 .md 文件然后告诉用户"已记下" | **违规，绝对禁止自动落盘** |
| 边界 | 用户说"记一下"但没有给标题，AI 追问标题后再写 | 合规，确认包含必要参数 |

### 规则 3：外部同步必须二次确认

推送 GitHub Issue 是可选的外部同步操作，需要独立于本地写入的第二次确认。

| 类型 | 场景 | 判定 |
|------|------|------|
| 正例 | 本地写入完成后询问"要同步到 GitHub Issue 吗？[Y/n]"，用户选 Y 后推送 | 合规 |
| 反例 | 创建本地 idea 后自动同步 Issue 不询问 | **违规** |
| 边界 | 用户首次选择 N（不同步），后续用 `/didea push --id N` 手动触发时仍需确认 | 合规，每次推送都需确认 |

## 六个硬入口动作

每个动作定义输入、执行行为、输出和前置条件。

### 1. create — 创建想法

| 项目 | 说明 |
|------|------|
| **输入** | `--title "想法标题"` + 可选 `--body "正文内容"` + 可选 `--tags "tag1,tag2"` |
| **行为** | 在 `.diwu/ideas/` 下生成 `{title}.md`，frontmatter 含自增 ID、时间戳、状态=idea |
| **输出** | 创建成功提示（含 id 和文件路径） |
| **前置条件** | 标题非空；`.diwu/` 目录存在 |

**frontmatter 必填字段**：

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `id` | int | auto (max+1) | 自增 ID，程序化引用用 |
| `status` | string | `"idea"` | `idea` \| `refined` \| `archived` |
| `created_at` | string | ISO 8601 now | 创建时间 |
| `updated_at` | string | ISO 8601 now | 最后更新时间 |
| `source_session` | string | `""` 或当前 session | 来源 session 文件名 |
| `github_issue_url` | string | `""` | 同步后的 issue URL |
| `linked_task_ids` | array | `[]` | 关联的 dtask id 列表 |
| `tags` | array | `[]` | 用户标签 |

**正文最小区块模板**：

```markdown
## 描述
（用户输入的想法正文）

## Why now
（为什么现在值得关注——可选）

## Next candidate action
（下一步可能做什么——可选，留白也合法）
```

### 2. list — 列出想法

| 项目 | 说明 |
|------|------|
| **输入** | 无参数，可选 `--status idea\|refined\|archived` 过滤 |
| **行为** | 扫描 `.diwu/ideas/*.md`，解析 frontmatter 输出摘要表 |
| **输出** | id / title / status / updated_at 摘要表，按 id 升序 |
| **前置条件** | 无（空目录输出空表） |

### 3. show — 查看想法详情

| 项目 | 说明 |
|------|------|
| **输入** | `--id N` |
| **行为** | 读取指定 idea 的完整内容（frontmatter + 正文） |
| **输出** | 完整 markdown 内容展示 |
| **前置条件** | id 对应的文件存在 |

### 4. refine — 补充完善想法

| 项目 | 说明 |
|------|------|
| **输入** | `--id N` + `--content "补充内容"` 或进入对话式补充 |
| **行为** | 在 idea 正文追加内容，更新 `updated_at` 和 `status`（可升级为 refined） |
| **输出** | 更新确认（含变更摘要） |
| **前置条件** | id 存在且 status != archived |

### 5. archive — 归档想法

| 项目 | 说明 |
|------|------|
| **输入** | `--id N` |
| **行为** | 将 idea 的 `status` 改为 `archived`，更新 `updated_at` |
| **输出** | 归档确认 |
| **前置条件** | id 存在；归档后不可 refine（只可 show 或 unarchive 如实现） |

### 6. push — 推送到下游

| 项目 | 说明 |
|------|------|
| **输入** | `--id N` + `--to dpth\|dref\|dprd\|dtask\|github` |
| **行为** | 根据 `--to` 目标执行对应交接逻辑（链式调用或回交建议） |
| **输出** | 交接结果（建议内容或链式调用结果） |
| **前置条件** | id 存在且正文非空；目标下游可用 |

详见下方「下游接口契约」章节。

## 下游接口契约

push 动作将 idea 推向四个下游能力。每种下游定义触发条件、传递数据、调用方式和预期产物。

### push → dpth（产品方向判断）

| 项目 | 说明 |
|------|------|
| **触发条件** | 想法需要判断"值不值得做"/"是不是真需求"/"方向对不对" |
| **传递数据** | idea #N 的完整正文（描述 + Why now + Next candidate action） |
| **调用方式** | 链式 `/didea push --id N --to dpth` 直接进入 dpth 诊断流程；或回交输出结构化建议引导用户手动 `/dpth` |
| **预期产物** | dpth 诊断结论（有立场判断 + 红旗信号 + 下一步动作）；回写 `linked_task_ids` 如产生关联 task |

### push → dref（需求细化）

| 项目 | 说明 |
|------|------|
| **触发条件** | 想法已经过初步判断（status=refined），需要拆成可执行清单 |
| **传递数据** | idea #N 正文 + dpth（如有）的诊断结论作为上下文 |
| **调用方式** | 链式 `/didea push --id N --to dref` 进入 dref 细化流程；或回交建议 |
| **预期产物** | dref 可执行检查清单；idea status 更新为 refined |

### push → dprd（产品论证 / PRD）

| 项目 | 说明 |
|------|------|
| **触发条件** | 想法足够成熟，值得正式写成 PRD（通常是大方向/核心功能） |
| **传递数据** | idea #N 完整内容 + dpth 诊断 + dref 清单（如有） |
| **调用方式** | 链式 `/didea push --id N --to dprd` 进入 dprd 论证流程；或回交建议 |
| **预期产物** | dprd PRD 文档；回写 `linked_task_ids` |

### push → dtask（任务化）

| 项目 | 说明 |
|------|------|
| **触发条件** | 想法已经 refine 过（有方案初稿），可以直接拆分为可追踪任务 |
| **传递数据** | idea #N 的 Next candidate action 区块 + refine 后的方案初稿 |
| **调用方式** | 链式 `/didea push --id N --to dtask` 进入 dtask 任务规划；或回交建议 |
| **预期产物** | dtask.json 新增条目；回写 `linked_task_ids`；task done 后 idea 可 archive |

### push → github（GitHub Issue 同步）

| 项目 | 说明 |
|------|------|
| **触发条件** | 用户希望将 idea 同步到 GitHub Issue（协作/公开追踪） |
| **传递数据** | idea #N 标题 + 正文（描述区块） |
| **调用方式** | 调用 `gh issue create` 创建 issue；**必须二次确认**后才执行 |
| **预期产物** | GitHub Issue URL；回写 frontmatter `github_issue_url` 字段 |

## 容器结构定义

### 目录位置

```
.diwu/
└── ideas/                    # 想法容器目录
    ├── Agent家族扩展.md       # 文件名 = 用户语言标题
    ├── 清理dadr残留.md
    └── ...
```

### 文件命名规则

- **格式**：`{用户输入的标题}.md`
- **不加前缀**：不用 `idea-NNN-` 前缀（那是机器噪音，人不需要）
- **清理规则**：去除文件系统非法字符（`/\:*?"<>|`），超长标题截断至 80 字符
- **自增 ID**：仅存在于 frontmatter 的 `id` 字段中，用于程序化引用和排序

### ID 自增策略

1. 扫描 `.diwu/ideas/*.md` 所有文件
2. 解析每个文件的 frontmatter `id` 字段
3. 取最大值 + 1 作为新 ID
4. 目录为空时从 1 开始

### frontmatter 完整字段规格

```yaml
---
id: 3                          # int, 必填, 自增
status: idea                   # string, 必填, enum: idea|refined|archived
created_at: 2026-05-08T00:13:45   # string, 必填, ISO 8601
updated_at: 2026-05-08T00:13:45   # string, 必填, ISO 8601
source_session: session-2026-05-08-001345.md  # string, 可选, 来源 session
github_issue_url: ""           # string, 可选, GitHub Issue URL
linked_task_ids: []            # array[int], 可选, 关联的 dtask id
tags: [agent, architecture]    # array[string], 可选, 用户标签
---
```

### 正文模板（最小区块）

```markdown
## 描述
（用户输入的想法正文，支持任意长度 markdown。这是唯一必填区块。）

## Why now
（为什么现在值得关注？什么时候变的紧迫？——可选，留白合法）

## Next candidate action
（下一步可能做什么？最想先验证哪条假设？——可选，留白合法）
```

## 反模式清单

| # | 反模式 | 为什么错 | 正确做法 |
|---|--------|---------|---------|
| 1 | **替代 dpth 做产品判断** | didea 是容器不是诊断器，不该输出"这个方向靠谱吗"的结论 | 挂住想法后 push → dpth 做判断 |
| 2 | **自动落盘不确认** | 违反动作门控规则 2，用户可能只是随口一说 | 始终等待明确确认后才写 `.diwu/ideas/` |
| 3 | **把外部同步当主路径** | 本地容器是主路径，GitHub Issue 是可选辅助 | 先写本地，再询问是否同步；不同步也完全合法 |
| 4 | **承载 drun 执行逻辑** | didea 不走 drun 执行链，不做代码修改/测试/验证 | 纯想法捕获+连接器，执行交给下游 |
| 5 | **改 rules 真相源** | didea 的产物是 ideas/ 下的 .md 文件，不是 rules/ 下的规则 | rules 真相源重组属于 Issue#6（PR5+ 范围） |
| 6 | **把想法当任务直接推进** | 想法和任务是两种不同粒度的实体，需要经过 refine 收敛 | idea → (refine) → (dref/dprd) → dtask，逐层收敛不可跳过 |
