## 不可协商规则

- 必须在用户明确确认后才落盘写入 `.diwu/ideas/`，禁止自动落盘
- 禁止对想法做诊断判断或细化拆解——didea 只负责挂住和连接下游
- 禁止代替 dpth/dref/dprd/dtask 执行其专属职责
- 想法文件必须包含 YAML frontmatter + markdown 正文，frontmatter 格式必须合法

# didea

想法的挂载点、持久化容器、下游能力连接器。**不是诊断器，不是清单生成器，不是 PRD 工厂。**

---

## 定位：第 0 层入口容器层

```
灵感 / 对话片段
  ↓
didea（挂载到 .diwu/ideas/）
  ↓
后续可进入 dpth(判断) / dref(细化) / dprd(论证) / dtask(任务化)
```

didea 只负责一件事：**让想法不丢失，并为它找到下一步的去向。**

### 与兄弟 skill 的界定

| skill | didea 只做 | 不做（留给对应 skill） |
|-------|-----------|---------------------|
| **dpth** | 挂住想法，判断入口 | 产品诊断（dpth） |
| **dref** | 挂住想法，后接细化 | 清单细化（dref） |
| **dprd** | 挂住想法，连接下游 | 完整 PRD 论证（dprd） |
| **dtask** | 想法 refine 后转任务 | 任务定义与执行（dtask+drun） |
| **architect** | 容器层 | 技术设计（architect） |

## 双入口模型

### 软入口（SKILL.md 自然识别）

在对话中识别到「这是一个值得挂起来的 idea」时：

1. **主动提议**：「这听起来像一个值得记录的想法，要不要挂到 `.diwu/ideas/`？」
2. **解释去向**：挂入后可以接 `dpth` 做判断、`dref` 做细化、`dprd` 写 PRD、`dtask` 拆任务
3. **等待确认**——**不自动落盘**

#### 识别信号

| 信号模式 | 典型话术 | 触发条件 |
|---------|---------|---------|
| 突然冒出新方向 | "对了，其实还可以做 XXX" | 对话中途插入无关新方向 |
| 明确要求暂存 | "这个先记下来"、"先挂着" | 显式暂存意图 |
| 未来式表达 | "以后可能再做"、"等有空了再考虑" | 时间不确定但认为有价值 |
| 灵感爆发式 | "我有个想法！"、"突然想到" | 高能量但未收敛的灵感

#### 提议话术模板

```
[捕捉到信号] → 「这听起来像是一个值得记录的想法。
 要不要我先把它挂到 .diwu/ideas/？
 挂进去之后随时可以用 /didea list 查看，
 也可以后续推给 dpth 做方向判断或 dref 做细化。」
```

#### 引导追问三句

在识别信号→用户确认写入后，写入前必须先追问以下三句（除非用户已提供）：

1. **标题确认**：「这个想法标题用『XXX』可以吗？」→ 用户确认或纠正
2. **场景展开**：「能展开一句，说说什么场景下需要这个吗？」→ 确保描述区块不是空壳
3. **紧迫性判断**：「为什么现在关注这个？是有什么新变化吗？」→ 填充 Why now

三句都拿到有意义的回复后，才能写入 `.diwu/ideas/`。至少拿到标题确认 + 场景展开（即描述非空），Why now 用户说"没有特别原因"也可接受（写入"暂无"）。

### 硬入口（commands/didea.md 显式操作）

用户明确调用 `/didea <动作>` 时的稳定操作入口。见下方「五个硬入口动作」章节。

## 动作门控：四条硬规则

> 这是 didea 的安全底线。违反任何一条都是越界。

### 规则 1：识别可以自然发生

软入口的识别是持续监听行为，不需要用户显式触发。

正例：用户说"对了还可以做个 XX"，提议挂入。反例：用户没表达想法，AI 主动问"有没有想法要记？"。边界：用户在讨论任务时顺带提"以后扩展"，温和提议即可。

### 规则 2：本地写入必须用户确认

无论软入口还是硬入口，写入 `.diwu/ideas/` 前必须获得用户明确确认。

正例：用户回复"好，记下来"后写入。反例：AI 直接创建 .md 文件然后说"已记下"——**绝对禁止**。边界：用户说"记一下"但缺标题，追问后再写。

### 规则 3：正文最低内容门槛

create 动作的「描述」区块不能是纯占位符（"ddd"/"test"/"."/"..." 等），AI 必须追问后才写入。

正例：用户给了标题+一句话描述，追问 Why now 后写入；或用户给完整三段直接写入。反例：只给标题、正文留白/占位符，AI 不追问直接写入——**违规，创建空壳**。边界：用户说"先记下来下午再补"，写入占位模板但标注「待补充」并提醒。合规，用户明确意图优先。


## 五个硬入口动作

- **create**: `--title` + 可选 `--body --tags` → 生成 `{title}.md`。标题非空。字段规格详见「容器结构定义」。创建前过追问三句（§双入口模型）。
- **list**: 扫描 ideas/*.md 解析 frontmatter 输出摘要表。空目录输出空表。
- **show**: `--id N` → 读取完整内容展示。
- **refine**: `--id N --content` 或对话 → 更新正文+updated_at。需 id 存在且非 archived。
- **archive**: `--id N` → 归档。归档后不可 refine（可 show 或 unarchive）。

## 容器结构定义

### 目录位置

`.diwu/ideas/{用户输入的标题}.md`（不加编号前缀，去非法字符，80 字符截断）。

自增 ID 仅存于 frontmatter `id` 字段：扫描现有文件取 max+1，空目录从 1 开始。

### frontmatter 完整字段规格

```yaml
---
id: 3                          # int, 必填, 自增
created_at: 2026-05-08T00:13:45   # string, 必填, ISO 8601
updated_at: 2026-05-08T00:13:45   # string, 必填, ISO 8601
source_session: session-2026-05-08-001345.md  # string, 可选, 来源 session
github_issue_url: ""           # string, 可选, GitHub Issue URL
linked_task_ids: []            # array[int], 可选, 关联的 dtask id
tags: [agent, architecture]    # array[string], 可选, 用户标签
---
```

### 正文模板（最小区块）

```markdown
## 描述
（用户输入的想法正文，支持任意长度 markdown。这是唯一必填区块。）

## Why now
（为什么现在值得关注？什么时候变的紧迫？——可选，留白合法）

## Next candidate action
（下一步可能做什么？最想先验证哪条假设？——可选，留白合法）
```
