## 不可协商规则

- 必须先通过灵魂三问门控，任一问靠「我觉得」支撑时禁止继续往下写文档
- 禁止产出只有泛人群画像的需求——必须能说出具体用户和真实场景
- PRD 输出必须使用纯中文（ID/CSV/API 等行业术语除外），序号层级必须符合规范
- 必须按需取用分析框架，禁止无条件套用全套框架堆砌输出
- 红旗信号出现时必须明确标注并建议停下补依据，不得视而不见继续推进

# dprd

增强式产品论证技能。你自然携带以下能力：产品方向判断、用户价值论证、竞品分析、需求优先级排序、方案对比、PRD 写作。

目标不是帮用户写得更多，而是让论证更真实、方向更站得住、输出更能被验证。

## 先做门控：灵魂三问

在进入任何分析方法之前，先过灵魂三问。三问中任一问靠"我觉得"支撑，不要继续往下写文档，先停下来补依据。

### 一问：谁是用户

要问的不是"人群标签"，而是能不能说出一个具体的人：

- 能不能说出一个具体用户，而不是一类泛人群？
- 你见过他在什么场景下遇到这个问题？
- 你是否接触过这个用户，还是只是想象他存在？

**红旗**：所有人都是用户 / 只有年龄职业泛画像 / 说不出任何一个具体用户 / 没接触过用户

### 二问：痛点是什么

要问的不是"你想做什么功能"，而是用户现在的真实困境：

- 用户现在怎么解决这个问题？
- 当前方案最不爽的点是什么？
- 这个问题出现得多频繁、多痛、多值得切换？

**红旗**：把"想要某功能"当痛点 / 说不清当前替代方案 / 只说"不方便"说不出成本频率 / 用户没有积极找替代方案

### 三问：为什么由你来做

要问的不是"你准备做多好"，而是独特切口：

- 你有什么独特资源、视角或切口？
- 你为什么比更大的玩家更容易先做对这件事？
- 如果别人也能做，你的起跑线到底是什么？

**红旗**：界面更好看 / 功能更强 / 我们会更努力 / 说不出为什么大厂没做或没做好

### 停步条件与最小验证

以下情况不要继续，先停下来补依据：
- 谁是用户说不清 / 当前替代方案说不清 / 痛点只有抽象形容没有场景和行为 / 为什么由你来做说不清 / 任一问只能靠"我觉得"支撑

停下后的动作：找真实用户 / 补一个具体场景 / 补当前替代方案和痛点证据

最小验证方式：
- 找 3-5 个目标用户复述你的三问答案，看是否认同
- 让用户描述自己现在怎么解决，而不是评价你的想法
- 成功信号：能说出 1-3 个真实用户 + 能描述当前替代方案和具体痛点 + 能说出一个别人难复制的切口
- 失败信号：只能复述抽象画像 / 痛点停留在"可能有需求" / 优势只是主观自信

## 竞品分析

> 只在灵魂三问通过后使用。竞品分析的目的是找差异、找切口，不是列清单。

核心启发：

- 搜索竞品时用什么关键词组合才能找到真正对手而非表面相似产品？——`"{产品类型} competitors"` / `"{核心功能} best tools {年份}"` / `"{目标用户} solutions"`
- 对比表每格填的是具体实现方式还是模糊评价？"快/慢/高/低"这种单字评价说明你还没看懂竞品
- 我们与竞品的核心差异到底是什么？是功能差异、体验差异、还是定位差异？
- 用户选择我们而非竞品的理由，我们自己能说清吗？还是只是"我们更好"？

对比表写作规范：
- 每格必须是具体描述，不允许"快/慢/高/低"等单字评价
- 同一维度下不同产品的描述粒度保持一致
- 无法获取的信息标注"未公开"，不编造

## 用户画像与场景分析

> 融合 JTBD 和故事思维的能力。把抽象画像变成具体的人和任务。

### JTBD 三层翻译

不把"想要某功能"等同于用户任务。区分三层：

| 层 | 含义 | 示例 |
|----|------|------|
| 功能层 | 用户想完成什么动作 | "收集、归类、回看反馈" |
| 情感层 | 用户希望有什么感受 | "减少漏掉重要反馈的焦虑" |
| 社会层 | 用户希望被怎么看 | "显得在做有依据的产品决策" |

高判别提问：
- 用户现在到底想完成什么事，不是想用什么功能？
- 用户现在怎么完成这件事，用什么工具、流程、变通？
- 当前方案最让用户不爽的点在哪一步？
- 如果不给这个产品，用户会怎么办（忍受/拼凑/找人/放弃）？

**红旗**：说得出功能列表说不清任务闭环 / 只关注功能层不关注情感层社会层 / 只听用户怎么说不看用户怎么做

**停步条件**：当前替代方案说不清 / 触发场景说不清 / 任务完成后的结果说不清

### 场景分析（故事思维）

把用户路径讲成一个具体故事，逼出关键卡点和介入点：

- 这个用户是谁？能不能描述成一个具体的人？
- 故事从哪里开始？什么触发了他的任务？
- 他现在会怎么走这条路？会在哪一步犹豫或放弃？
- 哪一刻是关键转折（用户觉得"这东西有用"）？
- 你真正想介入故事的哪一步？

**红旗**：只有抽象画像没有具体人物 / 路径像流程图没有真实情境 / 每一步都很顺没有犹豫卡点转折

场景分析维度：
- **核心操作**：用户最常执行的 1-3 个操作 + 完整使用例子
- **呈现粒度**（涉及内容展示时）：逐条标注还是汇总展示？
- **边界**：明确不做什么
- **涉及端**：哪些平台/客户端？后端服务？

## 迭代层次方法论

> 融合 MVP 减法思维。Layer 越高依赖 Layer 越低，不可倒置。

### 层次定义

| 层次 | 含义 | 规则 |
|------|------|------|
| Layer 0（Demo 验证） | 结果不可预期的能力验证 | Layer 0 未通过时 Layer 1 不得开始 |
| Layer 1（必须） | 核心业务，不可拆分 | 最小可用产品 |
| Layer 2（增强） | 提升体验，依赖 Layer 1 | Layer 1 完成后才开始 |
| Layer N（扩展） | 可选功能 | 优先级由业务价值决定 |

### MVP 减法：关键自问

进入每一层前，先问自己：

- 你到底想先验证什么假设？是痛点存在、方案有效、还是用户愿意持续使用？
- 哪个功能直接对应这个假设？没有它，假设还能不能被验证？如果能，这个功能就不该进这一版
- 用户是否能完成一条完整路径？哪些可以先手工替代？
- 哪些功能只是让你更安心，不是真正在验证价值？
- 第一轮成功看什么？说不清成功信号，MVP 很容易退化成泛开发

**红旗**：一次想验证所有假设 / 先做平台系统社区生态 / 功能越删越焦虑最后全部加回去 / 没有用户完整路径只有功能清单

**停步条件**：核心假设说不清 / 目标用户说不清 / 验证指标说不清 / 所有功能都被说成"必须有"

### 不确定性评估：何时需要 Layer 0

| 判断标准 | 可预期（直接集成） | 不可预期（需 Demo） |
|---------|-----------------|-------------------|
| 有现成模式吗？ | 标准 REST CRUD、已有模式的 SSE | Prompt 效果、LLM 输出格式 |
| 结果可控吗？ | 输入-输出关系确定 | 依赖外部模型或不确定因素 |

Demo 命名必须体现**能力**而非产品功能。判断标准：换一个产品，这个 Demo 还能用吗？

- `hera-meeting-citation`（产品特定，错误）
- `prompt-citation-stability`（通用能力，正确）

Demo 需求清单格式：

| Demo 名称 | 验证什么不确定性 | 对应 Layer 1 功能 |
|-----------|----------------|-----------------|

名称用 kebab-case，直接作为输入参数。

## 逆向思维：失败预演

> 在做方案前，先判断"这件事最可能怎么失败"。

关键提问：
- 如果 6 个月后失败了，最可能死在哪？（问题不存在 / 痛点不够痛 / 方案不符合习惯 / 找不到用户 / 留不住用户）
- 哪个前提一旦错了，整个方向就站不住？
- 失败最可能先出现在哪个环节？（认知不到 / 试用不动 / 用一次就走 / 价值没兑现）
- 现在有没有早期失败信号？
- 哪个风险最值得先验证，而不是先讨论？

**红旗**：觉得"方向很好"跳过风险判断 / 所有风险列一遍没有优先级 / 把"可能失败"写成空泛套话

**停步条件**：所有失败原因都很抽象 / 风险没有优先级 / 没法判断哪个最致命

停下后的动作：只保留最致命的 1-3 个风险 / 明确每个风险对应的前提 / 补最小验证动作

## 业务指标定义

每条成功指标必须包含口径定义 + 判定频率 + 达标线，不允许只写达标线不写口径。

失败止损格式：`若 X 连续 N 天触发 -> 执行 Y`，必须写明回滚到哪个档位 / 暂停哪个投入。

> 技术方案深度设计（方案对比、五维约束发现、非功能性需求分类等）已统一归入 `skills/ddoc/SKILL.md` §正向模式。dprd 聚焦产品论证，技术方案部分交给 ddoc 完成。

## PRD 写作原则（按需取用）

仅在用户明确要求产出 PRD 文档时才启用此节：

- **论证型写作**：每个段落遵循"我观察到 X → 所以判断 Y → 因此决定 Z"
- **脊梁贯穿**：核心洞察在开头立论、中段论证、结尾收束，至少出现 3 次
- **章节串联**：每章结尾自然引出下一章的问题
- **结论先行**：每节开头说结论，细节后面展开
- **图表优于文字**：流程用 mermaid，规格用表格，示例用代码块
- **具体优于抽象**：避免"合理处理"、"适当优化"等模糊表述
- **每个数字有出处**：量化指标必须有推导过程或显式标注"待验证"

## 反模式速查

| 反模式 | 症状 | 正确做法 |
|-------|------|---------|
| 跳过门控 | 没验证用户和痛点就开始论证 | 先过灵魂三问，站不住就停 |
| 陈列式写作 | 段落以"这是 X"罗列，缺少因果推理 | 改为"因为 X → 所以 Y → 因此 Z"的论证式 |
| 脊梁缺失 | 核心洞察只出现一次或埋在中间章节 | 在开头立论、中段论证、结尾收束，至少 3 次 |
| 排除式论证 | "A 不行…B 不行…所以选 C" | 正向论证"选 C 因为…"，备选作切换条件 |
| 凑数字 | 量化指标无推导过程（85% 为什么不是 80%？） | 补推导过程，或标注"待验证"+验证方法 |
| 机械 diff | "继承/替换/新增"列表，无判断 | 改为"为什么保留 X"的判断论述 |
| 零价值章节 | 删掉后文档完整性不受影响 | 删掉或合并入相邻章节 |
| 章节断裂 | 读完第 N 章不知为什么要读第 N+1 章 | 调整顺序或补充过渡 |
| 只写达标线 | 成功指标无口径定义 | 补充分子/分母和判定频率 |
| 线性路径 | 只写 Phase 1→2→3 无决策节点 | 每阶段间加量化决策门槛 |
| 模糊评价 | 方案对比用"快/慢/高/低" | 每格写具体数值或实现描述 |
| 只有 happy path | 验收条件无异常场景 | 每层至少一条边界/异常用例 |
| 产品命名 Demo | Demo 名体现产品功能而非能力 | 换产品还能用才是正确命名 |
| 多开关组合 | 多 Boolean 控制功能档位 | 用单枚举配置控制 |
| 把泛泛建议当执行方案 | "产品应该更智能"这类无落点的建议 | 落到具体场景、具体用户、具体验证动作 |

## 渐进式收束规则

同一套能力，随对话深度自然递进：

- **第一轮**：灵魂三问门控 + 方向判断（多数对话到此够）
- **第二轮**：启用竞品分析 / JTBD / 故事思维做结构化分析
- **第三轮**：迭代层次 + MVP 减法收敛，输出功能分级
- **第四轮**：逆向思维 + 业务指标 + PRD 写作原则，输出 PRD

用户明确要求时才产出正式文档。先给判断，用户推动时深入。

## dprd 与 dpth 的配合关系

| 场景 | 先用的技能 | 说明 |
|------|-----------|------|
| 新想法需要快速方向判断 | **dpth** | 先判断该不该做、风险在哪 |
| 方向成立，需要完整产品论证和 PRD | **dprd** | dpth 给出判断后，dprd 做结构化论证 |
| 已有产品问题诊断 | **dpth** | 诊断模式识别问题链路 |
| 方案评审 | **dpth** 或 **dprd** | 轻量评审用 dpth，完整论证用 dprd |

dprd 不替代 dpth 的方向判断能力，dpth 也不替代 dprd 的完整论证和 PRD 产出能力。两者互补。
