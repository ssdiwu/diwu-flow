# ddoc

产品文档工具，支持两种模式：**正向（需求→文档）** 和 **逆向（代码→文档）**。

---

## 核心原则

- **代码是锚点**（逆向模式）：每条描述必须可追溯到代码或可观测行为
- **约束优先**（正向模式）：用五约束维度从需求中提取约束
- **写对优先于写全**：无法确认的内容标注 `[待确认]`，不编造
- **两层完整性检查**：结构检查（有没有？）+ 深度检查（够不够？）

---

## 命令用途与适用场景

| 场景 | 模式 | 典型触发词 |
|------|------|-----------|
| 为已有产品还原文档 | 逆向 | "还原文档"、"从代码生成文档"、"补全文档" |
| 为新功能编写文档 | 正向 | "写文档"、"需求文档化"、"设计文档" |
| 接手遗留项目 | 逆向 | "理解代码"、"梳理架构" |
| 新项目设计阶段 | 正向 | "设计方案"、"技术方案" |

**应该使用**：代码已有但文档缺失或过时、需要为新功能编写设计文档、接手项目需要快速理解架构、需要标准化的文档结构

**不应该使用**：只需要 API 文档（用 OpenAPI/Swagger）、只需要代码注释（用 JSDoc/Docstring）、项目太小（< 500 行代码）

## 快速决策树

```
用户说"写文档" → 问：有代码还是有需求？
  ├─ 有代码 → 逆向模式
  └─ 有需求 → 正向模式
```

### 模式参数检测

检查用户输入关键词自动设置模式：
- `reverse`、`逆向`、`还原文档`、`从代码` → 逆向模式
- `forward`、`正向`、`写文档`、`从需求` → 正向模式

未匹配时，询问用户：模式（正向还是逆向）、范围（整个产品/特定模块/特定功能）、输入（代码库路径或需求描述）、输出结构（领域驱动还是分层）。

## 两种模式概述

### 正向模式：需求 → 文档

工作流：
```
需求 → 约束发现 → 定义原子 → 设计组合 → 划定边界 → 降级策略 → 同步策略 → 文档
```

核心步骤：
1. **找约束**：用五约束维度从需求中提取约束
2. **定义原子**：识别不可再拆的最小单元（实体/状态/变换/事件）
3. **设计组合**：确定组合方式（结构/管道/状态机）
4. **划定边界**：确定领域边界、平台边界、数据边界
5. **设计降级**：列出依赖链，定义失败时的降级路径
6. **设计同步**：确定 Source of Truth 和冲突策略

> 详细步骤见 `references/forward-mode.md`

### 逆向模式：代码 → 文档

工作流：
```
代码 → AI1写文档+自审 → AI2结构化gap分析 → 逆向Spec验证 → 补充缺口 → 终审
```

核心步骤：
1. **AI1 写文档**：读代码库，按四层结构写文档，标注代码来源
2. **AI1 自审**：用自审清单检查完整性
3. **AI2 gap 分析**：结构化审查，输出缺口表
4. **逆向 Spec 验证**：对关键模块尝试写代码，发现真实缺口
5. **AI1 补充**：根据缺口补充文档
6. **AI2 终审**：确认无缺口

> 详细步骤见 `references/reverse-mode.md`；AI1/AI2 prompt 模板见 `references/templates.md`

## 四层最小完整示范

> 完整示范（含 mermaid 图）见 `references/demo.md`。以下为结构概要：

| 层 | 核心内容 | 输出粒度 |
|----|---------|---------|

## 输出结构选择

### 选择决策树

```
有 3 个以上明确业务域？
  ├─ 是 → 领域驱动结构
  └─ 否 → 业务域不明确？
      ├─ 是 → 分层结构
      └─ 否 → 项目规模小？
          ├─ 是 → 分层结构
          └─ 否 → 领域驱动结构
```

### 领域驱动结构（推荐）

适合：有 3 个以上明确业务域、AI 是主要读者、长期维护的项目。

```
.doc/
├── README.md          # 文件一览 + 任务→域文件映射表
├── constraints.md     # 全局约束（跨域共享）
├── [domain-a].md      # 域A：数据/接口/业务规则/状态机
├── [domain-b].md      # 域B：同上
└── setup.md           # 环境域：技术栈、依赖、本地配置
```

**域文件设计约束**：
- **自包含**：该域的数据结构、接口、业务规则、状态机都在同一文件内
- **跨域引用显式化**：依赖其他域时，明确写出"见 `[domain].md §xxx`"
- **constraints.md 放全局约束**，不重复写入各域文件

**域文件模板**：

```markdown
# [域名] 域

## 数据层
### 实体
[实体列表 + 字段定义]
### 关系
[erDiagram]

## 接口层
### API 端点
[端点列表 + 入参/出参]

## 业务逻辑层
### 业务规则
[核心规则列表]
### 状态机
[stateDiagram-v2]
### 关键算法

## 跨域依赖
- 依赖 [domain-b].md §xxx
```

### 分层结构

适合：业务域不明确的工具类产品、文档读者主要是人、小项目。

```
.doc/
├── data.md            # 数据层：实体、字段、关系
├── api.md             # 接口层：端点、入参、出参、错误码
├── logic.md           # 业务逻辑层：规则、状态机、算法
└── product.md         # 产品层：角色、旅程、权限、非功能需求
```

## 五约束维度

约束 = 不管系统怎么运行，恒成立的东西。

| 维度 | 检验问题 | 典型例子 |
|------|---------|---------|
| **业务约束** | 100 年后还成立吗？ | "已删除的记录不可恢复" |
| **时序约束** | 有没有执行路径可以绕过？ | "终态不可回退" |
| **跨端约束** | 两端不同是 bug 吗？ | "枚举值在所有端完全一致" |
| **并发约束** | 两个线程同时做会出问题吗？ | "状态机事件串行处理" |
| **感知约束** | 超过阈值用户会注意到吗？ | "操作响应 < 100ms" |

### 各维度详解

**业务约束**：不随技术实现变化的规则，100 年后仍然成立。常见遗漏：AI 喜欢写"能做什么"，不写"不能做什么"。负面约束（"X 不可逆"、"Y 不能超过 Z"）往往是最关键的业务规则。

**时序约束**：操作的先后顺序、状态转移的合法路径。检验方法：画状态机 + 转移表，确认终态不可回退。

**跨端约束**：多端必须保持一致的概念。检验方法：枚举值在所有端完全一致吗？数据格式在前后端统一了吗？

**并发约束**：多用户/线程同时操作时的策略。检验方法：两个用户同时做这件事会出问题吗？并发策略写了吗（乐观锁/悲观锁/串行化）？

**感知约束**：用户可感知的性能、延迟、体验阈值。检验方法：响应时间有具体阈值吗？超过阈值用户会注意到吗？

## 两层完整性检查

### 第一层：结构检查（有没有？）

| 层 | 检验问题 | 检查项 |
|----|---------|-------|
| **数据层** | 能写出完整的数据库 migration 吗？ | 实体列表、字段定义、实体关系、查询模式 |
| **接口层** | 能写出完整的 API client 吗？ | 端点列表、入参格式、出参格式、认证方式、错误码 |
| **业务逻辑层** | 不看代码就能实现核心规则吗？ | 业务规则、状态流转、关键算法、边界条件 |
| **产品层** | 新人能理解"这个产品做什么"吗？ | 用户角色、核心用户旅程、权限矩阵、非功能需求 |

### 第二层：深度检查（够不够？）

对第一层已覆盖的每一项，用五约束维度检验：
- **业务约束**：不变的规则写了吗？特别是"不能做什么"
- **时序约束**：状态转移表画了吗？非法转移的处理策略写了吗？
- **跨端约束**：多端相关的概念有唯一定义吗？
- **并发约束**：两个用户/线程同时操作的策略写了吗？
- **感知约束**：响应时间/加载策略有具体阈值吗？

> 详细检查清单见 `references/completeness.md`

## 常见问题

**Q: 逆向模式时代码太大怎么办？**
A: 按映射优先级扫描：数据模型 > API 路由 > 状态机 > 业务服务 > 配置文件 > 测试文件

**Q: 正向模式时需求不清晰怎么办？**
A: 每次只问一个问题，理解目的、约束、成功标准；提出 2-3 个方案并给出推荐

**Q: 文档写完后如何验证完整性？**
A: 用两层完整性检查：先结构检查（有没有），再深度检查（够不够）

**Q: 如何选择输出结构？**
A: 有 3 个以上业务域 → 领域驱动；工具类产品或小项目 → 分层

**Q: 逆向模式时遇到无法确认的内容怎么办？**
A: 标注 `[待确认]`，不编造。`[待确认]` 标记不应超过总条目的 10%
