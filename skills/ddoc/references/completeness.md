# 两层完整性检查详解

## 概述

完整性检查分两层：
1. **第一层：结构检查**（有没有？）
2. **第二层：深度检查**（够不够？）

不是 4×5=20 个格子每个都必须填满。有意识地检查过，标注"不适用"即可。

## 第一层：结构检查（有没有？）

按 8 类文档类型逐一检查"该有的核心要素是否齐全"。每类的检查项取自 `doc-types.md` 的"关键要素"列。

| 文档类型 | 检验问题 | 检查项 |
|---------|---------|-------|
| **T1 产品需求（PRD）** | 需要不需要做、边界在哪？ | 用户故事（角色+目标+动机）、约束矩阵、验收场景（GWT）、非功能需求 |
| **T2 设计规范** | 技术上怎么做？ | 约束表（五维）、原子分解、状态机+管道图、边界定义、降级+同步策略 |
| **T3 页面规格** | 每页怎么工作？ | 组件树（层级+父子）、交互状态（加载/空态/错误/正常/边界）、数据绑定（字段↔组件映射）、路由与导航 |
| **T4 数据模型** | 实体和关系是什么？ | 实体列表、字段定义（名/类型/约束）、实体关系图（erDiagram）、主键/外键/唯一约束、查询模式+索引建议 |
| **T5 API 契约** | API/命令/事件契约是什么？ | 端点列表（路径+HTTP方法）、入参格式、出参格式、认证方式、错误码与含义 |
| **T6 变更记录** | 改了什么、影响谁？ | 变更条目（新增/修改/废弃/删除）、影响范围（API/DB/配置）、迁移指南（前后兼容+回退）、版本号对应 |
| **T7 运维手册** | 怎么部署和排查？ | 部署拓扑、配置清单+环境变量、操作SOP、告警规则+故障排查流程 |
| **T8 知识问答** | 常见问题和概念怎么查？ | Q&A对、概念索引、交叉引用、来源标注 |

### T1 产品需求（PRD）检查项

- **用户故事**：角色+目标+动机都完整吗？
- **约束矩阵**：业务/时序/跨端/并发/感知五维约束都覆盖了吗？
- **验收场景**：GWT 格式且 Then 子句可断言吗？
- **非功能需求**：性能、安全、可用性要求写了吗？

### T2 设计规范检查项

- **约束表**：五维约束（业务/时序/跨端/并发/感知）是否逐条展开？
- **原子分解**：实体/状态/变换/事件是否定义完整？
- **状态机+管道图**：状态转移表和流程图画了吗？
- **边界定义**：领域/平台/时间三个维度的边界是否明确？
- **降级+同步策略**：依赖不可用时的降级方案和同步策略写了吗？

### T3 页面规格检查项

- **组件树**：组件层级和父子关系写了吗？
- **交互状态**：加载/空态/错误/正常/边界所有状态覆盖了吗？
- **数据绑定**：字段到组件的映射写了吗？
- **路由与导航**：页面入口、出口和导航路径写了吗？

### T4 数据模型检查项

- **实体列表**：所有核心实体都列出了吗？
- **字段定义**：每个实体的字段（名称/类型/约束）都写了吗？
- **实体关系**：实体之间的关系（1:1、1:N、N:M）画了吗（erDiagram）？
- **主键/外键/唯一约束**：ID、索引、关联字段的约束写了吗？
- **查询模式**：常见查询场景和索引建议列了吗？

### T5 API 契约检查项

- **端点列表**：所有端点（路径 + HTTP 方法 / CLI 命令 / 钩子事件）都列出了吗？
- **入参格式**：每个端点的请求体/查询参数/路径参数写了吗？
- **出参格式**：每个端点的响应体结构写了吗？
- **认证方式**：认证机制（JWT/OAuth/API Key）写了吗？
- **错误码**：错误码列表和含义写了吗？

### T6 变更记录检查项

- **变更条目**：新增/修改/废弃/删除四类变更都记录了类型和内容吗？
- **影响范围**：每个变更的影响域（API/DB/配置）标注了吗？
- **迁移指南**：前后兼容方案和回退步骤写了吗？
- **版本号对应**：变更与版本号的对应关系写了吗？

### T7 运维手册检查项

- **部署拓扑**：服务依赖关系、网络架构图画了吗？
- **配置清单**：所有配置项和环境变量写了吗？
- **操作 SOP**：启动/停止/扩容/回滚的操作步骤写了吗？
- **告警与故障排查**：告警规则和常见故障排查流程写了吗？

### T8 知识问答检查项

- **Q&A 对**：高频问题的答案包含代码示例或引用链接吗？
- **概念索引**：术语→定义映射表完整吗？
- **交叉引用**：是否链接到其他文档类型的相关章节？
- **来源标注**：每条知识的来源（代码/文档/经验）标注了吗？

## 第二层：深度检查（够不够？）

对第一层已覆盖的每一项，用五约束维度逐一检验：

### 业务约束检查

- 不变的规则写了吗？
- 特别是"不能做什么"（负面约束）写了吗？
- 这些规则 100 年后还成立吗？

### 时序约束检查

- 状态转移表画了吗？
- "先后顺序"写清了吗？
- 非法转移的处理策略写了吗？

### 跨端约束检查

- 多端相关的概念有唯一定义吗？
- 枚举值在所有端完全一致吗？
- 数据格式在所有端统一了吗？

### 并发约束检查

- 两个用户/线程同时操作的策略写了吗？
- 锁机制（乐观锁/悲观锁）写了吗？
- 并发冲突的解决策略写了吗？

### 感知约束检查

- 响应时间有具体阈值吗？
- 加载策略写了吗？
- 用户可感知的性能指标写了吗？

## 检查清单

### AI1 自审清单（逆向模式）

□ 每个数据实体是否有完整字段列表？
□ 每个 API 是否有入参和出参的具体格式？
□ 每条业务规则是否有对应的异常情况处理？
□ 每个用户流程是否覆盖了"失败路径"？
□ 是否有任何地方用了"等"、"类似"、"根据情况"这类模糊表达？
□ 所有有状态的实体是否都画了 stateDiagram-v2？
□ 并发场景是否被考虑？
□ [待确认] 标记是否只在真正无法确认时使用？
□ 核心业务流程是否有 flowchart？
□ 数据实体关系是否有 erDiagram？

对每个"否"，直接补充内容后再交付。
