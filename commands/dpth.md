# /dpth — 产品思维协作

## 模式路由

- 用户输入 `/dpth` → 先选模式（诊断/创始人/构建器），再进入模式内工作流

## Step 1：确认核心定位

无论上下文是否已有描述，先确认核心定位再继续。

- 若用户**尚未描述**，直接问以下三个问题（一次问完）：
  - 要判断什么？
  - 当前处于哪个阶段？（有想法 / 需求模糊 / 方案评审 / 已有产品诊断）
  - 最想得到什么类型的输出？

- 若用户**已有描述**，复述核心定位并请用户确认，确认后进入 Step 2。

## Step 2：不确定性门控

| 条件 | 路径 |
|------|------|
| 改动小 + 结果可预期 + 差异一句话说清 | **直接进入模式内提问** |
| 预期复杂 / 外部依赖 / 有契约约束 | **先写最小规格或探索验证** |

## Step 3：模式选择与执行

根据用户意图选择模式并执行完整工作流：

- **诊断模式**：已有产品/功能，判断健康度、找痛点、识别红旗信号
- **创始人模式**：从零开始，验证假设、定义用户、构建 MVP 叙事
- **构建器模式**：已有方向，收敛方案、排优先级、规避风险

每种模式的完整工作流（五步工作顺序、高判别问题清单、红旗信号、输出格式）详见 `skills/dpth/SKILL.md`。

## 不做的事

- 不替代 dref（需求细化）或 dprd（PRD 产出）
- 不替 dtask/drun 做任务管理或执行
- 不改 rules 真相源或写 recording
