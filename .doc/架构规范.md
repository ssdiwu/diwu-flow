> 本文件是 diwu-flow 的**设计文档真相源**（`.doc/` 层）。

## Part A — 能力架构层

### 六层模型总览

```
L0 入口容器层     didea          想法挂住，要不要追踪
L1 判断/收束层   dpth dref dprd 值不值得做，怎么想清楚
L2 下游扩展层    architect debugger 技术设计边界 + 异常排查
L3 协议层        handoff model   dtask↔drun 交接契约（rules/handoff.md）
L4 规则真相源层  rules/ (14 文件) 运行规则唯一真相源
L5 表层能力模型  Commands/Skills 分层、drun dual-entry、Persistence Policy
```

### 层间主链路

```
用户有想法
  │
  ├─→ didea（L0：挂住想法，可选持久化）
  │     │
  │     ├─→ dpth（L1：产品方向判断）
  │     ├─→ dref（L1：需求细化清单）
  │     ├─→ dprd（L1：PRD 产品论证）
  │     └─→ 直接跳到执行层
  │
  ├─→ dtask（L5：任务定义与状态机）
  │     │
  │     └─→ drun（L5：单任务执行器）
  │           │
  │           ├─→ explorer → implementer → verifier（Agent 执行闭环）
  │           ├─→ architect（L2：大改动审稿，按需介入）
  │           ├─→ debugger（L2：异常排查，按需介入）
  │           └─→ drec（Session 记录归档）
  │
  └─→ dloop（L5：cron 驱动循环，定时触发 /drun）
        └─→ dstop（终止循环）
```

### Agents 完整列表（5）

| Agent | 所在域 | 核心约束 |
|-------|--------|---------|
| **explorer** | 只读调查 | 不修改任何文件 |
| **implementer** | 代码实施 | 先读后写；JSON indent=2 |

| **architect** | 技术审稿 | 不改代码；只审 dtask 定义域 |
| **debugger** | 异常调查 | 不直接修代码；诊断后回交修复链 |

> 完整交接协议与退化路径见 `agents/README.md` 和 `rules/handoff.md`。

### drun 双入口模型

`drun` 不再只是 "dtask runner"，而是 **Task Contract Runner**：

```
来源 A：dtask 中的持久化任务（InSpec → claim → InProgress）
  ↓
drun 执行（完整 Preflight → S1-S4 → closeout → /drec）

来源 B：用户直接描述的简单任务（direct request）
  ↓
drun 执行（简化流程：快速判断 → 实施 → 验证 → /drec）
```

**硬规则**：一旦命中 architect 审查条件（API/字段契约变更 / 单任务 >2000 行 / 新增模块），必须升级回 dtask 路径。

### Persistence Policy（持久化策略）

| 策略值 | 含义 | 适用场景 |
|--------|------|---------|
| `none` | 纯对话收口，不落盘 | 一次性问答、简单确认 |
| `drec` | 只写 recording + commit | direct run 的常规收尾 |
| `dtask` | 纳入任务体系 | 复杂任务、需跨 session 继续 |
| `dtask + drec` | 纳入任务 + 记录本轮 | 最完整的持久化 |

### Command / Skill 生命周期规则

#### 新增条件
- 有明确的方法论体量（≥ 可独立成 SKILL.md）→ 新增 Skill + Command
- 是纯操作入口（如 didea 的 CRUD 动作）→ 仅新增 Command
- 只是导航/索引够用 → 只更新 README

#### 删除条件
- 能力已被其他 Skill 完全覆盖 → 删除或溶解为索引
- 长期无人使用且无独特方法论价值 → 删除

#### 溶解条件
- 原有能力被拆分合并到多个新 Skill → 旧条目保留为导航索引（如历史上的 dvfy）

### Hooks 实现链（8 事件）

> 详见 `rules/testing.md` 和 `hooks/hooks.json`。此处只列事件与数据流概要。

| # | 事件 | 关键写入目标 |
|---|------|-------------|
| 1 | SessionStart | scoped session 文件 + pitfalls 注入 |

### `.doc/` vs `rules/` vs `README` 边界

| 层 | 定位 | 承载 | 不承载 |
|----|------|------|--------|
| **`.doc/`** | 设计文档真相源 | 架构解释、设计原理、插件源码仓结构规范 | 运行规则正文、Skill 方法论步骤 |
| **`rules/`** | 运行规则真相源 | 必须遵守的协议、状态流转、验证标准 | 设计 rationale、插件源码仓结构 |
| **`README`** | 说明/导航层 | 这是什么、怎么上手、去哪看细节 | 新增规则正文、替代真相源 |

**关键原则**：
- `rules/file-layout.md` 定义的是**目标项目**的文件组织（`.diwu/` 结构），不讲插件源码仓自身结构
- 插件源码仓结构规范在本文件 Part B 中定义
- 各层 README 只做导航索引，不新增规则

### 为什么 `rules/file-layout.md` 不讲插件源码仓结构

`rules/file-layout.md` 会通过 `/dinit` 同步到**用户项目**的 `.claude/rules/` 目录。如果混入插件自身的目录结构（如 `skills/`、`agents/`、`hooks/`），会导致：
1. 用户项目的 rules 目录出现与自身无关的插件内部结构描述
2. 同步时可能覆盖用户自定义规则（`不在清单中的文件会被删除`）
3. 违反「rules/ 只回答目标项目里什么东西该放哪里」的单一职责

因此插件源码仓结构只在 `.doc/架构规范.md` Part B 中说明，不进入 rules 传播链。

## Part C — Hooks 设计原则

> 回答：hook 脚本的数据访问边界、session/task 级别划分、多窗口安全约束。
> 导航索引见 `hooks/README.md`；本文件承载设计原理与架构约束。

### 两维分类框架

每个 hook 函数沿两个维度分类：

| 维度 | 取值 | 含义 |
|------|------|------|
| **数据访问层** | L0 / L1 / L2 | 该函数触碰什么数据 |
| **功能职责** | init / validation / monitoring / recording / decision / cleanup | 该函数解决什么问题 |

#### 数据访问层定义

| 层 | 数据来源 | 跨窗口风险 | 典型操作 |
|----|---------|-----------|---------|
| **L0** | 仅 event payload（stdin JSON） | 无——每事件独立 | 解析 session_id、判断事件类型、格式化输出 |
| **L1** | 项目只读文件（dtask.toml、recording/ 等） | 低——读不冲突 | 检查任务状态、检测归档阈值 |
| **L2** | 运行态读写（dtask-state.toml） | 有——多窗口并发写同一文件 | 写 session_id、清除 owner、更新 loop 计数 |

### Session 级 vs Task 级边界

> **核心约束：Session 级 hook 不应查询 task 级状态来反查"当前 session 对应哪个 task"。**

Claude Code 支持多窗口并行，每个窗口是独立 session。task 级状态（dtask-state.toml 中的 owner、InProgress 标记）是全局共享的，不属于任何单一 session。

| 事件类型 | 级别 | 允许访问的 data | 禁止的操作 |
|---------|------|---------------|-----------|
| **SessionStart** | Session 级 | 写 runtime state（初始化） | 查询"当前 session 的 InProgress 任务" |
| **Stop** | Session 级 | L0 event 数据 + L1 项目只读 | 从 dtask-state.toml 反查 owner → task 映射 |
| **PreCompact** | Session 级 | L0 event 数据 | 同上 |
| **TaskCreated** | Task 级 | dtask.toml（校验） | 修改非 task 字段 |
| **TaskCompleted** | Task 级 | dtask-state.toml（清除 owner） | 修改其他 task 的状态 |

#### 违反案例：stop_decision.py 的 A 组逻辑（已裁定删除）

```
Stop 事件触发
  → resolve_session_inprogress_task() 读 dtask-state.toml
    → 找到 owner=session_id 的 InProgress 任务
      → 问题：Window B 自由聊天时，可能找到 Window A 正在执行的任务
        → Window B 被误判为"有任务在执行"，触发错误的续跑提醒或阶段报告
```

根因：**owner 字段标识的是"谁在跑这个 task"，不是"这个 session 在跑哪个 task"**。一对多关系导致反查不可靠。

正确做法：Stop hook 只做 session 级事情（提醒 recording、提醒 decisions、dirty 检测），task 级事情由 TaskCompleted 等各自的事件处理。

### 多窗口安全策略

| 数据访问层 | 安全策略 |
|-----------|---------|
| L0 | 天然安全——event payload 每窗口独立 |
| L1 | 读操作应有容错：文件不存在/格式异常时不阻塞，跳过该检查 |
| L2 | 写操作通过 runtime state 的 owner 机制做互斥；读操作同 L1 容错 |

### 缓存隔离

使用文件缓存的 hook（context_monitor、drift_detect_pre），缓存文件名必须包含 `{session_id}` 后缀，避免多窗口共享同一计数器。

### 命名约定

统一采用 `{event}_{action}.py` 模式：

- **event** = Claude Code hook 事件名（SessionStart / Stop / TaskCompleted / PreToolUse / ExitPlanMode / PreCompact）
- **action** = 该 hook 在此事件中的核心动作

例外：
- `_shared.py` — 公共工具模块（下划线前缀标记非 hook）
- `run_hook.py` — 基础设施 wrapper
