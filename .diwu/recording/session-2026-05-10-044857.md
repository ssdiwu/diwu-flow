### 本次踩坑/经验

- [读层现象] 把"AI 能理解复杂指令"等同于"AI 会可靠执行复杂流程" → 错误。设计了四维打分引擎（D6 初版），形式完备但执行时 AI 更可能倒推填分数而非真正逐项评估。这与 triggers 10% 召回率是同一个根因：**结构化匹配/评分流程对 AI 来说不是自然行为，查表和语义直觉才是**。修正：L-Skill 从"评分引擎"降级为"路由参考卡"——信号速查卡 + 文本匹配，让 AI 做它擅长的事（模式匹配）而不是强迫它做它不擅长的事（结构化评分）
- [验证误读] XML tag (`<routing-check>`) 假设有机器可读语义 → 经 explorer Agent 实证调查确认 Claude Code 对 XML tag 无任何特殊处理。优先级由物理位置决定（用户级 > 项目级 > hook 注入位置），不由标记格式决定。教训：任何关于平台行为的假设必须验证，不能从其他系统（如 HTML/XML 解析器）类推
- [分层未拆清] L-Rule vs L-Skill 的职责边界在讨论过程中逐步清晰 → 最终锁定"L-Rule=自动守卫(每条回复前), L-Skill=路由顾问(不确定时深度分析)"。教训：双轨设计的核心不是"两个都做"，而是明确"什么场景下用哪个"——如果边界模糊就会变成重复劳动或互相推诿。L-Rule 解决"别漏掉"(90%消息)，L-Skill 解决"不确定时问一下"(10%边界 case)
- [路由护栏契约] 同一根因以不同面具第三次出现：triggers 10% 召回(design缺陷) → didea 零触发(同上) → 四维打分引擎不可靠(同上)。本质都是"期望 AI 按预定义流程执行检测/匹配/评分"。教训：这个根因会在每次设计"AI 自动执行 X"的方案时重复出现，识别后应立即检查：这个流程 AI 会天然执行还是需要强制？
