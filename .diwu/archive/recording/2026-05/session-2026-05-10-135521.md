### 本次踩坑/经验

- [分层未拆清] 分析 dref 是否需要多视角时，第一反应是"和 dpth 一样加双模"→ 但拆解原子能力后发现 dref 的核心 R4（苏格拉底式提问）和多视角判断是完全不同的认知模式。多视角=同一问题不同角色各看一遍（适合判断类），苏格拉底法=引导用户逐步暴露隐含假设（适合收敛类）。前者受益于多视角，后者会被多视角破坏（三个 explorer 同时提问造成信息过载）。教训：**给技能加能力前必须先拆解其原子能力，看每个原子是否真的需要增强，而不是因为"隔壁加了所以我也要加"**
- [读层现象] dref 的 description 写着"先判断问题真假，再收敛为可执行清单"——"先判断真假"这半句暗示 dref 有独立判断职责，但实际上这个职责完全被 dpth 覆盖了（且 dpth Active 更强）。description 的措辞导致了功能越位的错觉。教训：**skill description 是契约声明，不是能力愿望清单；写了"我能做 X"就意味着使用者会期望你做 X，即使 X 不是你的核心价值**
- [路由护栏契约] 同一根因第五次面具出现：triggers 10% 召回(design 缺陷) → didea 零触发(同上) → 打分引擎不可靠(同上) → Issue 评论不同步(信息源过时) → **技能功能越位(description 声称了不属于它的能力)(本次)**。本质始终是同一个："声称能做的事"和"实际该做的事"之间的错位。防御措施扩展：不仅检查 AI 执行流程可靠性，还要检查 skill 自身的能力边界声明是否准确
